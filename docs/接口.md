## 1. ATE 输出格式（文档 2.2 / 六.1）

**文档要求：**  
`ate_predictions_{domain}_{model}.jsonl`，**每行一个句子对象**，含 `sentence_id`、`sentence`、`predicted_aspects[]`，且每项含 `term`、`start_token`、`end_token`、`confidence`。

**当前实现（`ate/ate_predict.py`）：**

- 写出的是**一个 JSON 数组**（`json.dump(results, ...)`），**不是 JSON Lines**。
- 每条记录是 **`{"sentence", "aspect"}`**，且**每个 aspect 一行**（同一句话会被拆成多行），没有 `sentence_id`。
- **没有** `start_token` / `end_token` / `confidence`。
- 默认输出文件名是 `ate_predictions.json`，与文档中的 `ate_predictions_{domain}_{model}.jsonl` 不一致。

```88:100:ate/ate_predict.py
    results = []
    for ex in ds:
        sentence = " ".join(ex["tokens"])
        aspects = predict_aspects(model, tokenizer, ex["tokens"], device)

        for asp in aspects:
            results.append({
                "sentence": sentence,
                "aspect": asp
            })

    with open(args.output, "w", encoding="utf-8") as f:
        json.dump(results, f, indent=2, ensure_ascii=False)
```

内部推理用的是 **word-level** 的 `tokens` 列表和 `word_ids`（与文档六.4「还原为 word-level」方向一致），但**没有把索引和置信度写进输出**。

## 2. ASC 推理接口（文档 六.2）

**文档要求：**  
`predict(sentence: str, aspect_term: str) -> str`，返回 `"positive"` / `"neutral"` / `"negative"`（或等价 batch）。

**当前实现（`asc/asc_evaluate.py`）：**

- `predict(model, tokenized_dataset, device, batch_size=32)`：入参是**已 tokenize 的数据集**，不是 `(sentence, aspect_term)` 两个字符串。

## 3. 数据划分（文档 六.3）

**文档要求：**  
统一使用 **SemEval 2014 官方 train/test**，不做额外 random split。

**当前实现：**

- **ATE（`ate/ate_prepare_data.py`）**：`Restaurants_Test.xml` 作为 **test**；训练集来自 `Restaurants_Train.xml` 再 **random shuffle 后 90% train / 10% val**。Test 与官方 test 一致，但 train/val 是随机切分。
- **ASC（`asc/asc_prepare_data.py`）**：只读 **`Restaurants_Train.xml`**，对全部 `(sentence, aspect, label)` 做 **80% / 10% / 10% 随机划分**，**没有使用官方 Restaurants 测试集 XML**。这与文档「同一套官方 train/test」及 ASC_Report 里写的「官方无测试集标签所以随机划分」一致，但**与需求文档六.3 冲突**——若全组以六.3 为准，需要全队统一改划分策略。

## 4. Tokenizer / 索引（文档 六.4）

**文档要求：** ATE 输出的 token index 与句子 **word-level** 对齐。

**当前：** ATE 数据与预测路径基于 **空格分词** 的 `tokens` 列表；`predict_aspects` 里用 `word_ids` 把子词对齐回词。逻辑上可以满足 word-level，但**导出格式里未包含** `start_token`/`end_token`，集成时无法只靠现有 JSON 恢复 span。

## 小结表（接口对齐情况）

| 接口项 | 文档约定 | 当前代码 |
|--------|----------|----------|
| ATE 中间文件格式 | JSONL，每句一行，含 span + confidence | 单文件 JSON 数组，每 aspect 一行，无 id/span/confidence |
| ASC 推理签名 | `predict(sentence, aspect_term) -> str` | 仅批量 `predict(model, tokenized_dataset, ...)` |
| 标签字符串 | positive / neutral / negative | 评估里 ID 映射为上述三字符串，一致 |
| Train/Test 划分 | 官方 split，不 random | ASC：train 内随机划分；ATE：test 为官方，train/val 随机 |
| 端到端输出 | `e2e_predictions_*.jsonl` | 无对应实现 |

若要「与上游确认接口」，可把上表作为差异清单：**上游需补齐 JSONL 字段与置信度（或你在集成脚本里从模型 logits 补全）**；**ASC 需暴露或约定一层 `(sentence, aspect) → str` 包装**；**全队需统一是否改用官方 test（及 ASC 如何从有标签数据构造 test）**，否则 gold aspect 上的 ASC 指标与跨域实验无法和需求文档一一对应。
